Introduction

This tutorial is just a revamped version of the tutorial for VB programmers aimed at including C++ programmers.

So you've installed the latest and greatest ActiveLock 3 release, and now you're wondering how the heck you're going use this thing, right? Well, read on, my friend. This tutorial was written just for you. We will walk through a step-by-step process of putting ActiveLock 3 to work in your VB / C++  application.

Prerequisites

This tutorial was created using ActiveLock 3.x. Therefore, please ensure that you have the same version (or later) installed on your system before you proceed any further. For the latest ActiveLock 3 release, check here.

The first step is to create 2 security keys for your application. The ActiveLock security is based on a public / private key encryption. Due to the fact that many developers contributed to ActiveLock you may be faced with numerous names for these keys, so as an example in the following, G Code is the private key and V Code is the public key. There are other names lurking about!

Step 1: VB and C++. Generating a Product Key for your application

  1. Start the generator application, alugen.exe.
  2. Enter your application name in the Name text box. e.g. MyApp
  3. Enter your application version in the Version text box. e.g. 1.0
  4. Click the Generate button to generate the Product Keys.
  5. When the Product VCode and GCode generation process is complete, click Add To Product List button to add/save the product into the product database, to be used later for license key generation.
  6. Make a note of the generated VCode. You will use this code later for the "SoftwareCode" within your application.


Note that because of its length, the complete code may not be fully visible in the text box. You'll need to double-click on the text box and do a Ctrl-C operation if you want to copy the entire code.

Step 2: VB. Adding ActiveLock 3 Object Reference to your VB Project

  1. Using ALUGEN to generate a Product Key for your application
  2. Open your .vbp project in VB, and then select References from the Project menu.
  3. Scroll down on the list until you see ActiveLock Object Library 3.0, then select by clicking on the empty checkbox beside it. Note: If you dont have that exact library then check you've installed the latest version of Activelock.

Step 2: C++. Adding ActiveLock 3 Object Reference to your VB Project

 

The equivalent of  VB of adding the Activelock3 Reference to your program is accomplished with the addition of the line

#import "C:\\windows\\system32\\ActiveLock3.4.dll"

The C++ programmer has a lot more to do than his VB brother, and in this respect the following files have been provided

  1. common.h,crc32static.h and crc32static.cpp determines the CRC value of files, in particular ActiveLock3.4.dll and alcrypto3.dll.
  2. Comutilfix.h fixes some faults present in some earlier forms of comutil.h, which may or may not be necessary still today.
  3. ConnectionAdvisor.h, ConnectionAdvisor.cpp, EventSink.h and EventSink.cpp are used to connect to the events within the ActiveLock dll
  4. ActiveLockUtil.h and ActiveLockUtil.cpp contain miscellaneous procedures
  5. ActiveLockMFC.h and ActiveLockMFC.cpp
    These files provide several functions:
    a. Conversion to and from COM structures to more normal C++ structures, eg BSTR to CString, SAFEARRAY to vector.
    b. More high level class tying in all the above providing an easy to use interface.

Step 4: Checking for a Valid License

OK. So now you've got ActiveLock successfully initialized. At some point in your application's lifetime, it will need to check for a valid license. This is accomplished by calling IActiveLock's Acquire() method. 

         25   ' Attempt to acquire a valid license token
         26   On Error GoTo ErrHandler
         27   ActiveLock.Acquire  ' Acquire will raise an error if no valid license exists.
         28   Exit Sub ' Exit Form_Load()
         29 ErrHandler:
         30   MsgBox "ActiveLock Error: " & Err.Description
         31 End Sub
 

When ActiveLock.Acquire() returns no error, it means you currently have a valid license. At that point you can continue to verify some other properties of the license such as RegisteredDate, RegisteredUser, UsedDays, ExpirationDate, etc... For further information on these properties, refer to the IActiveLock interface in the API Documentation.

Step 5: License Registration

Your application must provide a facility for your users to request license keys. It needs to tell your users what their installation code is. This is an ActiveLock-computed code that is unique for that user's machine alone. To keep it simple for this tutorial, I will simply use a TextBox on the TutorialApp Form to display the installation code. Let's also add a text box for specifying the registered user. This way, the registered user will now be part of the Installation Code, which means that the liberation will not specific to the machine, but it's also specific to a particular user. The code that re-computes the installation code might look like this: 

         32 ' Re-compute Installation Code when "Registered User" changes
         33 Private Sub txtUser_Change()
         34     txtInstCode = ActiveLock.InstallCode(txtUser)
         35 End Sub
 

Your users will then be instructed to contact you with this Installation Code in exchange for a liberation key to unlock the application. This contact can be done via either a telephone call or an email message. You can also add the capability to do instant activation by having your application connect directly to your web server, accessing an ASP on that server, which will use ALUGENLib installed there to generate the liberation key, and then return the key immediately for registration. But this is a topic for another tutorial :-). For the purpose of this tutorial, I will assume that your customer either calls or emails you the Installation Code.

So with the Installation Code in hand, you:

  1. Open up ALUGEN
  2. Go to the License Key Generator tab
  3. Select MyApp from the Product drop-down list
  4. Select a License Type. e.g. Periodic for a license that will expire after x days
  5. Set the Expiration Date if applicable
  6. Select the Registered Level
  7. Paste the Installation Code into its text box
  8. Click the Generate button
  9. In the Liberation File textbox, replace activelock.alb with myapp.all. Then click Save.

Congratulations! You've generated your first ActiveLock 3 liberation key.

Now all you have to do is give this myapp.all file to your user, and tell them to place in your application directory, and then restart your application. The rest will be taken care by ActiveLock.

Step 6: Let's Get A Little More Advanced

So far we've done the very basic steps required to get ActiveLock 3 up and running within your app. This will prevent the very novice of hackers from breaking your protection. But a mediocre hacker will attempt to break it by cracking the ActiveLock 3 dlls, and since ActiveLock is open-source, this can be quite easily done. Therefore, you'll need to employ a few extra stragedies :)

Handling ActiveLockEventNotifier_ValidateValue() Events to Do Some Customized Encryption

If you look inside the license file, you'll see this (note that the long keys have been partially chopped for better formatting within this document): 

[TestApp]
ProductVersion=1.0
LicenseType=1
LicenseClass=Single
Licensee=Evaluation User
RegisteredLevel=0: Level 0
MaxCount=1
LicenseKey=AAAAB3NzaC1yc2EAAACARuESlkdcig8MQDbE59HY3slGbfuGrfky0jfqKmgF8wLY9...
RegisteredDate=2005/7/1
LastUsed=2005/07/01 15:04:13
Hash1=c95c8a93a9856033aac8882e5585c9b2
Expiration=2005/7/2

You may have noticed the Hash1 license property. The value for this property is an MD5 hash value of the LastUsed property. LastUsed property is used to detect when the user has set their system clock backward in order to continue using the app after its license has expired. To defeat this detection, all the hacker has to do blank the LastUsed property, computes his own MD5 hash for an empty string, enters it into the Hash1 property, then he can continue using your application with no problem. To prevent this type of hack, ActiveLock 3 makes use of the ActiveLockEventNotifier_ValidateValue() event. This event allows you to encrypt the LastUsed value using your own encryption technique that only you know, then it will take your encrypted value, and computes an MD5 hash for it, then saves it as the hash1 value.

Here's an example of a simple encryption implementation: 

''
' Simple encryption of a string: simply multiply every character's ASCII code by 11.
'
46 Private Function Enc(strData As String) As String
47    Dim i&, n&
48    Dim sResult$
49    n = Len(strData)
50    Dim l As Long
51    For i = 1 To n
52        l = Asc(Mid$(strData, i, 1)) * 11
53        If sResult = "" Then
54            sResult = Hex(l)
55        Else
56            sResult = sResult & "." & Hex(l)
57        End If
58    Next i
59    Enc = sResult
60 End Function
 

Then, you handle ActiveLockEventNotifier_ValidateValue() event like this: 

61 Private Sub ActiveLockEventSink_ValidateValue(ByRef Value As String)
62    Value = Enc(Value)
63 End Sub

Authenticating ActiveLock 3 DLL To Detect Cracking

There's a saying that security people usually like to say: "a chain is only as strong as its weakest link". Because ActiveLock is open-source, and hence its implementation completely exposed to the public, there is no way for it to employ any "good" anti-cracking technique. In this sense, it can be your weakest link. Your application, on the other hand, is closed-source and therefore a little more difficult to crack.

Hackers can either crack the ActiveLock 3 DLL or use the source to build an imposter DLL that pretends to be the real ActiveLock and does anything the hacker wants it to, such as fooling your application into thinking that there is a valid license in the system when in fact there isn't. Therefore, your application must authenticate the ActiveLock 3 DLL before use, and one way to do it is to use the built-in CRC checksum facility that's provided by Windows. Here's the general process of how to use CRC checksum to authenticate the DLL.

  1. First, you compute the CRC checksum activelock3.dll and store it within your source code, possibly encrypted, as a private constant value.
  2. Somewhere within your application, typically in the initialization code and before initializing ActiveLock, compute the current CRC checksum of activelock3.dll and compare it against the saved checksum. If they don't match, then the DLL has been cracked.

You can find working code of how this is done by looking in the source code for the example application, ALTestApp, which is included in the ActiveLock 3 distribution.

Encrypt your SoftwareCode

You may have noticed that the Software Code is a humongously long text string that can be easily spotted by a Hex Editor and replaced by a hacker with their own Software Code. You should therefore attempt to hide this from the hacker by storing it in encrypted format and make your application decrypt it when needed.

Avoid using VB's builtin string functions to do your encryption

One of the tools usually employed by hackers is SmartCheck. This program can see all parameters passed to public DLL function calls of a running program. It is important to emphasize that only public function calls can be seen by SmartCheck; your private function parameters are safe from its prying eyes. Therefore, it is advisable that your encryption/decryption routines NOT use the VB builtin functions such as Mid, Left, InStr, etc... Instead, you should implement these functions (Mid, Left, InStr, etc...) by yourself, making them do hard-core byte array manipulation instead of regular string manipulation, for instance.

Trial Period Feature

The long awaited and anticipated feature of having a Trial Period in ActiveLock 3 is here. Below we will guide you through the basic steps to enabling this new feature.

In order to implement the Trial Feature into your code, you have to add code into your Form_Load event. Minimal code needed to activate the trial feature is marked with TRIAL FEATURE.

The GetTrial function in the Activelock3.Globals class does all the work for you. This call returns True or False depending whether the Trial Feature is active/valid or inactive/invalid.

The arguments for this function are as follows:

trialPeriodStatus = ActiveLock3.Globals.GetTrial(AppName, SlotName, SlotPassword, TrialType, TrialPeriod, StrMsg)

 AppName: Name of your application such as App.ProductName, App.Title or similar. Make sure that this is not an empty string otherwise a False will be returned.
 SlotName: Any name you choose to use for the trial feature. Must not be an empty string.
 SlotPassword: Any password you want. Must not be an empty string.
 TrialType: Trial Days=0; Trial Runs=1
 TrialPeriod: Number of Days for TrialType=0; Number of Runs for TrialType=1
 StrMsg: Message returned from Activelock. This is a string array that includes the registration status (for trial) and the number of days or runs used during the trial period. Split it using vbCrlf and process it.

Notes:
 Trial Feature is an optional feature. When you add it to your application, your compiled EXE will become a trial version of your application (i.e. MyAppWithTrialPeriod.EXE). If you do not add the Trial Code into your application, then the compiled exe will not have the trial feature (i.e. MyAppWithoutTrialPeriod.EXE). You can have either option but not both at the same time.
 Once the Trial Days/Runs is activated, the user can always register the product using the Generate button that you provide. You have to modify the Generate button Click code as follows to allow users register the application while the trial feature is still active:

Private Sub cmdReqGen_Click()
' Generate Request code to Lock
If MyActiveLock Is Nothing Then
    noTrialThisTime = True
    Form_Load
End If
txtRegStatus.Text = ""
txtUsedDays.Text = ""
txtReqCodeGen.Text = MyActiveLock.InstallationCode(txtUser)
End Sub

 Once an application is succesfully registered, the trial feature is killed and cannot be reactivated.
 In Version 3 of Activelock, the Acquire event accepts optional arguments. noTrialThisTime is a public boolean needed to handle a registration request initiated by the Generate button. You must adjust the code:

MyActiveLock.Acquire
as
If noTrialThisTime Then
    MyActiveLock.Acquire AppName, SlotName, SlotPassword
Else
    MyActiveLock.Acquire
End If

Register to Level X Feature

All you need to do to retrieve this information from your application is to read the RegisteredLevel property of the licence and activate/deactivate certain portions of your code accordingly. This is shown with the REGISTERED LEVEL FEATURE example above. You should also populate the cboRegisteredLevel combo box in Alugen to facilitate production of licenses for a given level of registration.

Note: Registered Level is an optional feature. If not set, the application will always register to level 0. Registered Level feature has no impact on your application unless you use it.
